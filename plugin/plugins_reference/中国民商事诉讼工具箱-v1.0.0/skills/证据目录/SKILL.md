# 证据目录

撰写民商事诉讼证据目录，按照要件攻防分析的结构对证据进行分组编排，生成规范的证据目录文档。

## 职责边界

> **本技能仅负责**：证据的逻辑排序、分组归类、证据目录表格的文本撰写。
>
> **本技能不负责**：证据文件的物理整理、重命名、编页码、装册等文件操作。

## 前置条件

根据主诉/被诉场景，所需材料不同：

### 主诉场景（原告举证）

| 序号 | 材料 | 说明 |
|------|------|------|
| 1 | 起诉状 | 已生成的起诉状（明确诉讼请求） |
| 2 | 案件事实梳理 | 经整理的时间线与关键事实 |
| 3 | 要件攻防分析 | 包含各要件及其对应的证据支撑 |
| 4 | 证据材料清单 | 用户手头实际拥有的证据材料列表 |

### 被诉场景（被告举证）

| 序号 | 材料 | 说明 |
|------|------|------|
| 1 | 答辩状 | 已生成的答辩状（明确抗辩策略） |
| 2 | 案件事实梳理 | 经整理的时间线与关键事实 |
| 3 | 要件攻防分析 | 包含各抗辩要件及其对应的证据支撑 |
| 4 | 证据材料清单 | 用户手头实际拥有的证据材料列表 |

如缺少上述材料，应提示用户先完成相应步骤。

## MCP 预检

> 预检协议详见 qoder.md「MCP 预检协议」。本技能使用后端：LAW.*（闸门阶段法条引用校验）。
> 不可用时降级标注：`[L4-法条待验证]`。

## 工作流程

### Step 1: 确定举证方向

1. 识别当前场景：主诉（原告举证）还是被诉（被告举证）
2. 明确举证目标：
   - 主诉：证明诉讼请求成立
   - 被诉：证明抗辩事由成立 / 反驳对方主张

### Step 2: 提取要件结构作为分组框架

1. 从要件攻防分析中提取要件结构
2. 将要件结构作为证据分组的依据
3. 确定各组的排列顺序：

**主诉分组典型顺序**：
- 第一组：主体资格证据（原被告身份、权利主体）
- 第二组起：按请求权构成要件依次排列
- 末组：损害赔偿计算证据（如有金钱请求）

**被诉分组典型顺序**：
- 第一组：程序性抗辩证据（如有）
- 第二组起：按抗辩要件依次排列
- 末组：反证/否认证据

### Step 3: 将证据材料对应到各组

1. 逐份证据分析其证明方向
2. 将证据归入对应的要件分组
3. 同一组内按以下优先级排序：
   - 直接证据优先于间接证据
   - 书证优先于其他证据类型
   - 时间顺序（同类证据按时间排列）
4. 对于一份证据可能证明多个要件的情况：
   - 归入最主要的证明方向所在的组
   - 在证明对象中可附注"同时证明……"

### Step 4: 编号与撰写证据目录

#### 编号规则

采用"**组号-序号**"格式：
- 第一组第一份证据：`1-1`
- 第一组第二份证据：`1-2`
- 第二组第一份证据：`2-1`
- 依此类推

#### 表格结构

| 列名 | 宽度占比 | 说明 |
|------|----------|------|
| 编号 | 10% | 组号-序号 |
| 证据名称 | 25% | 证据的规范名称 |
| 证据来源 | 15% | 证据的形成方式/获取途径 |
| 证明对象 | 50% | 该证据用于证明的具体事实 |

#### 证明对象撰写规范

- 以"证明……"开头
- 简明扼要，一般1-2句话
- 聚焦于该证据直接能证明的事实
- 避免过度延伸或主观评价
- 表述应与要件攻防分析中的论证相呼应

**示例**：
- 证明被告于2023年5月1日与原告签订《技术开发合同》，约定开发费用为人民币50万元
- 证明原告已按合同约定完成全部技术开发工作并交付成果
- 证明被告收到开发成果后未在约定期限内提出异议

### Step 5: 添加分组标题与组说明

在证据目录中为每组添加分组标题，简要说明该组证据的整体证明方向：

```markdown
## 第一组：主体资格证据
本组证据用于证明原、被告的诉讼主体资格及法律关系基础。

| 编号 | 证据名称 | 证据来源 | 证明对象 |
|------|----------|----------|----------|
| 1-1 | …… | …… | 证明…… |
| 1-2 | …… | …… | 证明…… |

## 第二组：合同成立与生效证据
本组证据用于证明原被告之间合同关系成立并生效。

| 编号 | 证据名称 | 证据来源 | 证明对象 |
|------|----------|----------|----------|
| 2-1 | …… | …… | 证明…… |
| 2-2 | …… | …… | 证明…… |
```

### Step 6: 输出文档（md 优先）

> **⛔ 交付前必过闸门**（见 qoder.md「法律核验闸门」）：核验对象为证据目录中法条引用。

**默认交付 Markdown**：核验放行后先产出 md 成品（写入输出目录并回链），随后询问用户「是否转 Word（Word 转换后端（DOCX.md_to_docx, preset=legal））」，确认才转 .docx。

> **输出文件命名**：`{案件简称}_证据目录_{日期}.md`。Word 版同名 `.docx`。

## 证据来源常见类型

| 来源类型 | 适用场景 | 示例表述 |
|----------|----------|----------|
| 双方签署 | 合同、协议 | "原被告双方签署" |
| 单方制作 | 函件、通知 | "原告/被告制作" |
| 第三方出具 | 鉴定报告、审计报告 | "[机构名称]出具" |
| 行政机关 | 登记证书、行政决定 | "[机关名称]颁发/作出" |
| 公证 | 公证书 | "[公证处名称]公证" |
| 网络取证 | 电子证据 | "网页截屏/区块链存证" |
| 当事人持有 | 原始凭证 | "原告/被告持有原件" |

## 质量检查清单

生成证据目录后，自查以下要点：

- [ ] 分组结构是否与要件攻防分析的要件体系一致
- [ ] 编号是否连续且格式统一（组号-序号）
- [ ] 每份证据是否有明确的证据名称
- [ ] 证据来源是否准确填写
- [ ] 证明对象是否以"证明……"开头
- [ ] 证明对象是否简明且聚焦
- [ ] 是否存在遗漏的关键证据
- [ ] 同组内排序是否合理（直接证据优先）
- [ ] 分组标题是否准确概括该组证明方向
- [ ] 各组之间逻辑递进是否清晰

## 注意事项

1. **与攻防分析对齐**：分组结构必须与要件攻防分析中的要件体系保持一致
2. **客观表述**：证明对象应当客观描述证据能证明的事实，避免主观评价
3. **不遗漏不多余**：用户提供的每份证据都应入目录；未实际持有的证据不编入
4. **实事求是**：如发现关键要件缺少证据支撑，应在该组备注"待补充"
5. **不涉及文件操作**：本技能不重命名文件、不编页码、不整理文件夹

## 下一步建议

证据目录完成后，根据案件进展可继续：
- **庭审准备** → `/庭审提纲` 基于证据清单生成庭审应对预案
- **质证准备** → `/质证意见` 对对方证据出具质证意见（被诉场景）
- **可视化** → `/诉讼可视化` 将证据结构生成可视化图表

## Verification

生成完成后确认：
1. 证据目录结构完整（分组标题 + 表格）
2. 分组逻辑与要件攻防分析一致
3. 编号规则统一（组号-序号格式）
4. 证明对象表述规范且与案件事实对应
5. Word文档成功生成
6. 未包含任何文件操作指令
