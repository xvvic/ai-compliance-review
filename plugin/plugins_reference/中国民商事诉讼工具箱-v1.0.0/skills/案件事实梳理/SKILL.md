# 案件事实梳理

从案件材料中客观中立地提取事实信息，产出**一份主文档 + 两张配图**：

- **主文档：案件事实梳理（Markdown）**——这是交付的核心文档，按大事记构建逻辑，依次包含三个部分：
  1. **法律关系分析图**——嵌入 HTML 图的链接 + 一段配套文字说明（主体构成、核心法律关系、争议关系）
  2. **法律事实时间轴图**——嵌入 HTML 图的链接
  3. **大事记**——结构化时间线表，含来源标注、重要性评级、缺口提示与版本号
- **配图一：法律关系分析图（单文件 HTML）**——按「两要素三步法」呈现主体与法律关系，由主文档第一部分链接引用
- **配图二：法律事实时间轴图（单文件 HTML）**——按「轴线要素 + 事件要素」呈现关键事实的时间脉络，由主文档第二部分链接引用

> **交付结构**：两张图各自是独立 HTML 文件（可在浏览器打开、便于分享），主文档 md 通过 `file://` 链接把它们串进来，再附大事记表格。用户打开主文档即可总览全局，点击链接查看大图。

> **可视化定位**：可视化图表通过图文结合把复杂法律关系、交易结构、时间逻辑直观化，便于法官快速理解案情、锁定核心证据。一份事实清晰、注明依据的关系图与时间图，能显著节省阅卷时间，并为当事人争取更有利的审理印象。绘图遵循「让人看懂、简化清晰、客观呈现」三项基本要求。

> **大事记定位**：事实按顺序发生，大事记是每个叙事依赖的骨架——代理词的事实部分、法律意见、庭前准备都建立其上。AI 擅长结构化提取，但**输入垃圾则输出垃圾**：大事记只与来源一样好，不发明来源中没有的事件。

## 适用场景

- 用户提供案件材料，要求梳理案件事实
- 需要从材料中提取法律关系和时间线/大事记
- 案件初始阶段的案情分析
- 跨 skill 联动中需要事实基础支撑

## 核心原则

### 1. 客观中立

- 不预设任何一方立场
- 不对事实做倾向性评价
- 区分「已证事实」与「待证事实」
- 对争议事实标注各方主张

### 2. 法律关系观察准确

- 准确识别主体身份和角色
- 权利义务关系定性准确（合同/侵权/物权/人格权等）
- 关注法律关系的产生、变更、消灭
- 区分基础法律关系和派生法律关系

### 3. 时间线不是流水账

- 只纳入对法律后果有影响的事件
- 关注法律时效的起算和中断
- 标注关键时间节点的法律意义
- 过滤无关的日常事务性信息

### 4. 来源可追溯

- 每个大事记条目必须标注信息来源
- 无法追溯到提取文件的条目须内联标注：`[联网检索——需复核]`、`[模型知识——需验证]`、`[用户提供]`
- **不得删除或压缩来源标签**

### 5. 材料存疑标注

- 对矛盾之处明确标注（不替当事人解决矛盾——见「本技能不做什么」）
- 对缺失环节提示补充（写入「缺口」区）
- 对模糊表述保留原文

## 工作流程

### 第零步：保密门禁（每次先运行）

大事记与图表均从文件中提取，文件可能包含保密或受保护信息。提取前先确认：

> 这些来源是否已经过保密筛选？
>
> - **A. 所有来源已清理**——已筛选过，提取时不加保密标记。
> - **B. 混合或尚未筛选**——提取并为敏感条目加标记。
> - **C. 中止——先筛选**——暂停技能，待用户筛选来源后重新运行。

### 第一步：接收材料与识别来源

根据用户提供的材料和说明，判断案件场景：

| 场景 | 特征 | 关注重点 |
|------|------|----------|
| 主诉场景 | 用户为原告/申请人 | 请求权基础、证据完整性 |
| 被诉场景 | 用户为被告/被申请人 | 抗辩事由、对方证据瑕疵 |
| 中立分析 | 未明确立场 | 双方权利义务全貌 |

**来源识别顺序**：

1. **用户提供的路径**——本次会话中提供的任何文件
2. **默认案件文件夹**——用户指定的案件工作目录
3. **声明的来源**——配置或对话中声明的来源
4. **询问**——来源看起来不足时，主动提示用户补充

**材料接收清单：** 起诉状/答辩状/仲裁申请书、合同及附件、往来函件/通知、转账凭证/发票、会议纪要/聊天记录、鉴定报告/审计报告、判决书/裁定书（如有前序程序）、其他相关材料。

对于无法访问的已声明来源，在输出的「缺口」区中明确命名，不得静默跳过。

### 第二步：客观中立抽取事实

#### 2.1 主体识别

```
主体信息模板：
- 名称：（全称 + 简称）
- 类型：自然人 / 法人 / 非法人组织 / 国家机关
- 角色：原告 / 被告 / 第三人 / 其他
- 关键属性：（住所地、法定代表人、资质等与案件相关的属性）
```

#### 2.2 法律关系识别

```
法律关系模板：
- 主体A → 主体B
- 关系类型：合同关系 / 侵权关系 / 物权关系 / 担保关系 / 劳动关系 / 其他
- 具体定性：（如：买卖合同 / 借款合同 / 产品责任 / 抵押权等）
- 权利内容：（主体A对主体B享有的权利）
- 义务内容：（主体A对主体B负有的义务）
- 关系状态：存续 / 已终止 / 争议中
- 依据：（合同条款 / 法律规定 / 事实行为）
```

#### 2.3 事实层次分类

| 层次 | 定义 | 标注方式 |
|------|------|----------|
| 确定事实 | 双方认可或有充分证据证明 | 直接陈述 |
| 争议事实 | 双方说法不一致 | 标注「原告主张：... / 被告主张：...」 |
| 待查事实 | 证据不足无法判断 | 标注「待补充：...」 |

### 第三步：事件提取、去重与重要性标记

#### 3.1 事件提取

对每份材料，识别带日期的事件，参考句式：

- **邮件/函件：** `[日期] [发送人] 告知 [收件人] [主题/内容]`
- **会议：** `[日期] [参加人] 就 [主题] 开会`
- **决定：** `[日期] [决策人] 决定 [什么]`
- **诉讼文件：** `[日期] [当事人] 提交 [起诉状/答辩状/上诉状]`
- **外部事件：** `[日期] [事情发生]`（合同签署、付款、产品交付、监管行动等）

纳入大事记的事件应满足以下至少一项：法律关系变动、时效影响（起算/中断/中止）、关键证据形成、程序节点、损害发生或扩大、重要意思表示的发出或到达。

#### 3.2 去重

同一事件可能出现在多份文件中——这是**一个有多个来源的事件**。合并为一条，引用所有来源。

#### 3.3 按案件理论标记重要性（侧重点框架）

同一事件因执业者是在证明主张还是反驳主张而具有不同重要性：

- **原告/主张方（进攻框架）**——🔴 标记*确立*请求权要件的事件（责任、因果关系、损害、通知）、*关闭*对方将试图打开的缺口的事件，或*启动*诉讼时效的事件；🟡 标记支持主张但可被质疑的事件；⚪ 是背景。
- **被告/抗辩方（防御框架）**——🔴 标记*打破*请求权要件的事件（因果关系断裂、通知缺失、依赖缺失）、*开启*诉讼时效或管辖权抗辩的事件，或*支持*积极抗辩（免责、弃权、过错相抵）的事件；🟡 标记削弱对方叙事的事件；⚪ 是背景。

> **纪律**：300 条条目中有 300 条 🔴 等于没有标签。只为真正能移动事实认定者的事件保留 🔴。

#### 3.4 排除标准

与案件无关的日常事务、重复性常规操作（除非某次有特殊意义）、纯粹内部讨论（除非构成意思表示）一律不纳入；无法确定时间的事件移入「缺口」区单独列出。

### 第四步：生成法律关系分析图（HTML）

确定法律关系是制定诉讼策略的第一步。法律关系分析图按「两要素 + 三步法」绘制，**输出为单文件 HTML**，可直接在浏览器打开、便于分享。

#### 4.1 主体要素（第一步）

将所有法律主体（名称或代号）定点。主体范围不限于当事人；**同一主体只出现一次**；按角色归类以便配色。

#### 4.2 关系要素（第二步）

梳理主体、主要事实与各方法律关系。关注并存的多层关系（如借贷中的借款/保证/抵押；建设工程的承包/转包/分包）；将有联系的主体用连线串联，并用简短文字解释连线含义；区分基础关系与派生关系。

#### 4.3 连线与文字说明（第三步）

兼顾法律事实与时间顺序连接；减少交叉与曲折；图文结合（关系类型、时间、状态、金额、份额标注在连线/节点上）；复杂图可拆分。

#### 4.4 生成 HTML

构建一个**自包含单文件 HTML**（`法律关系分析图_{案件名}.html`），结构如下：

- 顶部：案件名称标题 + 一行说明（构建日期、视角/场景）
- 主体配色图例：原告蓝 `#dae8fc` / 被告红 `#f8cecc` / 法院绿 `#d5e8d4` / 第三方紫 `#e1d5e7`
- 关系图主体：用内嵌 [Mermaid](https://cdn.jsdelivr.net/npm/mermaid/dist/mermaid.min.js) 渲染 `graph LR`（左右结构，与套件一致），节点用 `subgraph` 按角色分组、连线 label 写关系文字、争议/待证关系用虚线 `-.->`、已终止关系标灰
- 底部：争议关系与关键金额/份额的文字注解

HTML 模板骨架（占位符替换为实际内容）：

### 第五步：生成法律事实时间图（HTML）

时间要素是案件核心要素——事件时间点或先后顺序常与争议焦点直接相关（违约金起算、占用费时段、合同解除节点、诉讼时效判定）。法律事实时间图按「轴线要素 + 事件要素」绘制，**输出为纯 HTML/CSS 单文件**（无外部依赖，最稳健）。

#### 5.1 轴线要素

一条竖轴代表案件时间轴，各时间点按先后排列；**时间点精确到具体年月日**，不省略。

#### 5.2 事件要素

每个时间点对应一个事件卡片，简洁概述「该时间点发生了什么」；卡片格式统一；**关键事实（🔴）用醒目强调色**区分，使受众一眼锁定要害。

#### 5.3 生成 HTML

构建自包含 HTML（`法律事实时间图_{案件名}.html`），用纯 CSS 竖向时间轴呈现，关键节点高亮。模板骨架：

### 第六步：组装案件事实梳理主文档（Markdown）

将前两步生成的两张 HTML 图与筛选去重后的大事记，组装为**一份主文档** `案件事实梳理_{案件名}.md`。这是交付给用户的核心文件，依次包含三个部分：

1. **法律关系分析图**——用 `file://` 链接引用第四步生成的 `法律关系分析图_{案件名}.html`，并在链接下方写一段配套文字说明：本案主体构成、核心法律关系定性、争议关系与关键金额/份额。文字说明须客观，不作法律适用结论。
2. **法律事实时间轴图**——用 `file://` 链接引用第五步生成的 `法律事实时间图_{案件名}.html`。
3. **大事记**——筛选去重后的时间线表（含来源标注）+ 关键事件（🔴）+ 缺口 + 时效争议提示 + 版本号。

链接写法：使用图 HTML 文件的绝对路径，如 `[查看法律关系分析图](file:///绝对路径/法律关系分析图_{案件名}.html)`。

若已有先行版本：**版本号递增**，向用户呈现 diff 摘要。

最终确定前确认：

> 这是我构建的内容。请浏览 🔴 条目——有无我判定错误的地方？

## 输出产物

本 skill 完成后产出**一份主文档 + 两张配图**，主文档通过链接把两张图串起来：

### 主文档：案件事实梳理（Markdown）

- 格式：Markdown 文档（大事记结构），**交付的核心文件**
- 结构：元数据头 +「一、法律关系分析图」（HTML 链接 + 文字说明）+「二、法律事实时间轴图」（HTML 链接）+「三、大事记」（时间线表 + 关键事件 🔴 + 缺口 + 时效争议提示 + 版本号）
- 输出：写入 `.md` 文件，以 `file://` 链接交付；可版本递增
- 要点：两张图的链接为可点击的 `file://` 绝对路径；关系图配套文字说明客观、不作法律结论

### 配图一：法律关系分析图（HTML）

- 格式：单文件 HTML（内嵌 Mermaid `graph LR` + 角色配色图例）
- 方法：两要素（主体 + 关系）三步法
- 输出：写入 `.html` 文件，由主文档第一部分链接引用；离线降级为内嵌 SVG 或调用 `/诉讼可视化` 出独立 `.html`
- 内容：主体不重复、连线少交叉、图文结合、争议关系虚线

### 配图二：法律事实时间轴图（HTML）

- 格式：单文件 HTML（纯 HTML/CSS 竖向时间轴，无外部依赖）

## 常见案件类型的梳理要点

### 合同纠纷
- 重点：合同成立→生效→履行→违约→解除 的完整链条
- 时间线关注：履行期限、通知到达日、宽限期
- 法律关系重点：主合同关系、从合同关系（担保等）

### 侵权纠纷
- 重点：侵权行为→损害结果→因果关系→过错
- 时间线关注：侵权行为发生日、损害发现日、持续侵权期间
- 法律关系重点：加害行为主体、共同侵权、补充责任

### 公司纠纷
- 重点：股东关系、决议效力、高管义务
- 时间线关注：决议日期、工商变更日期、知情日期
- 法律关系重点：股权结构、实际控制关系、关联交易

### 劳动纠纷
- 重点：入职→履行→变更→解除/终止 的完整链条
- 时间线关注：入职日、合同期限、解除通知日、仲裁时效
- 法律关系重点：劳动关系确认、劳务派遣关系

## 本技能不做什么

- **不解决矛盾**：当两份文件就同一事件说法不同时，两个条目都放入并标记，由律师判断取舍。
- **不发明来源中没有的事件**：不在文件中的，就不在产物中（联网/模型推断须打标签）。
- **不保证完整性**：产物仅与来源一样好，缺口必须如实写入「缺口」区。
- **不替代法律判断**：仅提供客观事实整理与可视化，不作法律适用结论。

---

## 注意事项

1. **材料引用准确**：所有事实必须有材料依据，不得推测；推断须打来源标签。
2. **动态更新**：用户补充材料后更新主文档与两张图，主文档大事记版本号递增。
3. **保密意识**：每次先过保密门禁；提醒用户注意材料中的敏感信息。
4. **篇幅适度**：大事记时间线建议控制在 30 行以内，过长应分段或附索引。
